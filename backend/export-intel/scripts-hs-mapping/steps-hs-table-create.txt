INSERT INTO hs_code_reference (hs_code)
SELECT DISTINCT hs_code
FROM market_intelligence;



The codes like 20130 and 30617 are actually HS codes with a leading zero that got dropped when the data was ingested earlier.

Examples:

Correct HS6	Stored in DB
020130	20130
030617	30617

This happens because somewhere during ingestion they were treated as numbers instead of strings, which removed the leading zero.

The good news: this is easy and safe to fix.

Step 1 — Identify All Broken HS Codes

Run:

SELECT hs_code
FROM hs_code_reference
WHERE LENGTH(hs_code) < 6;

This will show all HS codes missing leading zeros.

Step 2 — Fix Them (Pad With Leading Zeros)

Run:

UPDATE hs_code_reference
SET hs_code = LPAD(hs_code, 6, '0')
WHERE LENGTH(hs_code) < 6;

What this does:

20130  → 020130
30617  → 030617
Step 3 — Verify the Fix

Run again:

SELECT hs_code, LENGTH(hs_code)
FROM hs_code_reference
LIMIT 10;

Expected now:

020130 | 6
030617 | 6
100630 | 6
120510 | 6
Step 4 — Fix the Same Issue in market_intelligence

Since market_intelligence.hs_code is the column we join on, we must fix it there too.

Run:

UPDATE market_intelligence
SET hs_code = LPAD(hs_code, 6, '0')
WHERE LENGTH(hs_code) < 6;
Step 5 — Verify Join Still Works

Run:

SELECT mi.hs_code, ref.product_description
FROM market_intelligence mi
LEFT JOIN hs_code_reference ref
ON mi.hs_code = ref.hs_code
LIMIT 10;

Everything should still join correctly.

Step 6 — Prevent This From Happening Again

When loading HS codes in scripts, always enforce string formatting.

Example in Python:

hs_code = str(hs_code).zfill(6)

This guarantees:

30617 -> 030617
